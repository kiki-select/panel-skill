# 命名 / 空间 / description 治理规范

建 panel/dashboard 是往 BI 平台写**持久化资产**，会被团队长期看到。脏看板的危害比一次性错 CSV 大得多
（长期误导决策、污染检索）。本技能对治理强约束。

## 1. 防 prod 污染

- 在正式服（如 app_id=42）建报表前，先确认是**正式沉淀**还是**测试**。
- **测试/AI 临时产物**：标题加显式前缀（`AI测试-`），用完即删，别留在正式空间。
- **正式 AI 沉淀**：标题加 `AI-` 前缀，标明来源，便于审计与区分人工报表。
- 新建看板默认落在**当前用户私有空间**，不会直接进公开业务空间 —— 要进公开空间需人工搬运。

## 2. 命名规范（必须与现有看板目录统一）

**权威来源**：看板目录与命名规范以飞书 wiki 为准 —— https://sofunny.feishu.cn/wiki/EHXXwKlPSiksmMkaO7qc6HBNnVd
（该 wiki 需登录查看；新建/调整命名前应先对照它，下面是据线上目录归纳的落地规则。）

panel `title` / dashboard `name` 要**够具体、可检索、防重名**（重名会被自动加 `(1)`）。

### 命名要呼应 FunnyDB 的空间 / 文件夹层级

`dashboards/search` 返回的空间(space) → 看板(dashboard)层级就是业务分类骨架。命名时让报表能**对号入座**到对应业务域，便于检索与归位。app42 现有主要空间：

| 空间 | 覆盖业务域 |
|---|---|
| 香肠整体数据 | 活跃 / 留存 / 付费 / 游戏参与度 / 模式参与 / 摸金匹配 … |
| 玩法系统 | 各玩法模式（经典/摸金/团竞/狼人…）、身份卡、枪械、载具、BR 等（含文件夹如「经典派对相关」） |
| 摸金派对 | 摸金活跃/流失/养成/匹配/段位/物资/战斗/关卡… |
| 活跃系统 | 逐周活跃、新手引导、段位、各类活动 |
| 付费系统 | 流水、各付费模块、季票、商城 |
| 外围系统 | 社交、好友、战队、设置、登录 |
| 安全策划 / 安全运营 | 举报、外挂、封禁、检测 |

### 命名结构

- 推荐结构：`[前缀] 业务域-指标-粒度[-限定]`
  - `AI-经典派对-逐日登录账号数`
  - `AI-经典派对-身份卡选择占比-按段位`
  - `AI-摸金-撤离成功率-按地图`
- **业务域用现有目录里的标准叫法**（如「经典派对」「摸金」「BR」），不要自造同义词，避免和人工报表割裂。
- 禁止：`测试`、`新报表`、`panel1` 这类无信息名。

> 待办：飞书 wiki 里若有更细的命名字典（前缀规范、指标命名表、空间归属规则），同步补进本文件并固化为硬规则。

## 3. description 必填（与 ds 看板检索闭环）

**这是本技能的硬约束**。ds-skill 检索看板已改为「description-优先」（读「主要内容/核心指标」判口径来提命中率）。
我们建的 panel 如果没写 description，就是给下游检索挖坑。

格式（与现有人工报表一致）：

```
主要内容：<这个报表算什么、用来回答什么业务问题、口径要点>。
核心指标：<逗号分隔的指标/维度列名>
```

示例：

```
主要内容：按身份卡统计经典模式局内更换次数和占比，用于看玩家身份卡选择偏好。
核心指标：身份卡、更换次数、占比
```

要求：
- 「主要内容」写清**口径**（哪个事件、什么过滤、什么粒度），不是只抄标题。
- 「核心指标」列出实际**列名**，与 SQL 输出别名对齐。
- 有特殊口径（如"更换≠开局选择""已排除AI/封禁"）必须在主要内容里点明。

## 4. 建库前必校验（铁律）

**未经校验的 query 禁止建 panel。** 校验 = ds-skill 的探查 + 抽样校验流程跑通，或脚本建完
`--verify` 拉数确认能出数且数值合理。建一个错口径的常驻看板 = 长期事故。
