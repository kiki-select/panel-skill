# 命名 / 空间 / description 治理规范

建 panel/dashboard 是往 BI 平台写**持久化资产**，会被团队长期看到。脏看板的危害比一次性错 CSV 大得多
（长期误导决策、污染检索）。本技能对治理强约束。

## 1. 防 prod 污染

- 在正式服（如 app_id=42）建报表前，先确认是**正式沉淀**还是**测试**。
- **测试/AI 临时产物**：标题加显式前缀（`AI测试-`），用完即删，别留在正式空间。
- **正式沉淀**：遵循下方目录标准命名（`主题-指标-粒度`），**不强加 `AI-` 前缀**——以便直接归入现有目录、与人工报表统一；来源通过 description 标注即可。
- 新建看板默认落在**当前用户私有空间**，不会直接进公开业务空间 —— 要进公开空间需人工搬运。

`一级目录 → 二级目录 → 三级目录(看板) → 具体面板(panel)`。前三级是业务分类骨架，第四级才是我们建的 panel。命名时让 panel 能**对号入座**到对应业务域。

| 一级目录(部分) | 覆盖 |

| 香肠整体数据 | 活跃/新增/留存/付费/参与度/模式参与/摸金匹配/摸金OKR |

| 活跃系统 / 付费系统 / 外围系统 / 安全运营 | 见目录 |

### 命名结构（按「调整后名称」列归纳）

**`<主题/模块>-<指标/对象>[-<粒度/限定>]`**，分隔符统一用 `-`。真实示例：

| 调整后名称 | 拆解 |
|---|---|
| `活跃-同时在线-5分钟趋势` | 主题 活跃 / 对象 同时在线 / 粒度 5分钟趋势 |
| `活跃-分时DAU-逐时累计` | |
| `留存-新增账号留存率` | 无粒度后缀也可 |
| `付费-大盘概况-逐日` | |
| `模式参与-经典派对-逐日` | |
| `摸金经济-货币产销-活跃玩家` | 末段也可是人群/细分 |
| `摸金战斗-地图KD分布` | |

### 标准词汇（不要自造同义词）

| 类别 | 标准叫法（用这些） | 禁止 |
|---|---|---|
| 模式 | **经典派对 / 摸金派对 / 街机派对** | 经典模式 / 撤离模式 / 摸金 / 街机模式 |
| 时间粒度 | 逐日 / 逐时累计 / 5分钟趋势 / 按周 | 每天 / 按日 / 趋势图 |
| 人群 | 活跃玩家 / 新进玩家 / 回流玩家 | 老玩家 / 新用户 |

- 禁止：`测试`、`新报表`、`panel1` 这类无信息名。
- 拿不准主题/模块叫法时，去「总目录」找最接近的现有 panel 对齐。

## 3. description 必填（与 ds 看板检索闭环）

**这是本技能的硬约束**。ds-skill 检索看板已改为「description-优先」（读「主要内容/核心指标」判口径来提命中率）。
我们建的 panel 如果没写 description，就是给下游检索挖坑。

格式（与现有人工报表一致）：

```
主要内容：<这个报表算什么、用来回答什么业务问题、口径要点>。
核心指标：<逗号分隔的指标/维度列名>
```

示例：

```
主要内容：按身份卡统计经典模式局内更换次数和占比，用于看玩家身份卡选择偏好。
核心指标：身份卡、更换次数、占比
```

要求：
- 「主要内容」写清**口径**（哪个事件、什么过滤、什么粒度），不是只抄标题。
- 「核心指标」列出实际**列名**，与 SQL 输出别名对齐。
- 有特殊口径（如"更换≠开局选择""已排除AI/封禁"）必须在主要内容里点明。

## 4. 列标题用中文（展示友好）

panel 是**实际要展示**的面板，SQL 输出的每一列都要用**中文别名**，方便看板直接读。

- 每个 select 列都给中文别名：`count(distinct pid) as "登录账号数"`、`` `#dt` as "日期" ``（中文别名用双引号）。
- **不要留英文/原始字段名当列头**（如 `pid`、`cnt`、`item_name`）。
- 列名与 description 的「核心指标」对齐。
- **格式（单位、round、百分比、小数位）先不用管** —— 由人工在看板里调整。我们只保证列名是中文、口径正确。

## 5. 建库前必校验（铁律）

**未经校验的 query 禁止建 panel。** 校验 = ds-skill 的探查 + 抽样校验流程跑通，或脚本建完
`--verify` 拉数确认能出数且数值合理。建一个错口径的常驻看板 = 长期事故。
